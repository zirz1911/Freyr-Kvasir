## Honest Feedback

**1. Background agent permission design is friction.** The agent completed the cognitive work — researched the current issue, drafted all corrections, wrote the full body — but couldn't execute because it lacked Bash tool permission. So the user got a notification saying "task complete" and then had to watch me re-read the issue and rewrite the body anyway. From Lokkji's perspective, the background agent added latency without saving effort. The permission model needs to be set at launch time: if a task needs `gh issue edit`, grant it explicitly when spawning.

**2. Context compaction mid-task is jarring.** The summary was good — it correctly identified the 8 missing JS lines and even pre-wrote them. But there's still a cognitive seam: the user sees a break, has to trust the summary is accurate, and can't verify without reading the generated code. For short completions like this, it worked. For larger incomplete tasks, the compaction risk is higher.

**3. Issue #2 rewrite required reading the old body first.** I couldn't just apply diffs — I had to fetch the full 260-line body, reconstruct the corrected version, write it to a temp file, then `--body-file`. A cleaner pattern would be targeted section replacement. `gh issue edit` doesn't support partial section updates though, so this is a tool limitation, not a process mistake. Still, the round-trip added ~5 minutes that felt avoidable.

## Lessons Learned

- Background agents need explicit tool grants at spawn time — if the task involves `gh` commands, include Bash in the delegation or the "completion" is fake
- `gh issue edit --body-file` is clean and reliable; write body to temp file first when the content is long
- Detail panels / drawers are a good UX pattern for analysis-heavy presentations — surface depth without cluttering the main flow
