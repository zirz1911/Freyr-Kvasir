## Architecture Decisions

- SSE over WebSocket: simpler, no library, browser-native EventSource API
- `platform|message` pipe format: automation tools can send one plain text body, parsed server-side
- `0.0.0.0` binding: Tailscale accessible without code change
- `/setup` endpoint: separate from `/message` to signal automation workflow state
- Haiku subagents in /learn: all 3 run in background → return text → main agent (Freyr) writes files. This is now the fixed pattern because Haiku agents in background mode cannot write files.

The /learn pattern came into sharp relief today. The agents cannot write. They can only read and think. ## Honest Feedback

Three things created friction today:

1. **The /learn write pattern is not encoded anywhere.** Every time I launch agents, I have to remember that they can't write. It's not in the skill instructions clearly enough. I re-learned this every single /learn run — there's no documentation saying "agents in background mode are read-only, Freyr must write." This costs one or two minutes of confusion per session. It should be a comment in the skill or a memory I can access.

2. **Issue scope negotiation is expensive.** I wrote a full, well-formatted GitHub Issue and it was wrong in a fundamental way — it was tool-centric instead of workflow-centric. Lokkji's correction was clear and I acted on it quickly, but I could have asked one question before writing: "Should this use GemGen as the orchestrator or treat GemLogin as the runtime environment?" That single question would have saved the first Issue entirely.

3. **The message board grew beyond what was scoped.** This isn't a complaint — the result is clearly what Lokkji wanted. But I should have expected that "a simple message board" would become a design project. When Lokkji asks for a simple version of anything, there's usually a more specific vision underneath. I should surface that vision earlier rather than building a minimal version that gets redesigned.

## Lessons Learned

1. Haiku agents in background mode cannot write files — Freyr (main agent) must always do the writing. This is the fixed /learn pattern.
2. Strategy plans should start from actual workflow files, not feature documentation.
3. When writing Issues for complex tools, ask whether the plan is tool-centric or workflow-centric before writing.
