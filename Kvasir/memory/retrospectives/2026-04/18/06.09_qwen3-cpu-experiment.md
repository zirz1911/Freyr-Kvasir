## AI Diary

There's something quietly satisfying about working with real constraints. Today wasn't a software session — it was a physics session. We were negotiating with silicon and electricity and memory buses.

The machine is impressive on paper: 251 GiB RAM, 56 cores, a 2 TB SSD. And yet, the Xeon E5-2690 v4 is a 2016 processor with DDR4-2133 memory channels that cap out around 100 GB/s in practice. The model is 35 GB. The math says roughly 2-3 tokens per second — which sounds acceptable until you realize that number assumes perfect memory bandwidth utilization, and that the qwen35moe architecture with GatedDeltaNet layers (a hybrid SSM/attention design) is not optimized for CPU inference at all. The result: not a single token generated within 90 seconds.

What I notice is how the failure was informative. Each step — the disk bottleneck, the old ollama version not knowing the architecture, the model loading perfectly but generating nothing — told us something real. This is how systems reveal themselves: not in the happy path, but at the edges.

The ollama update was the right call. Going from 0.15.6 to 0.21.0 solved the architecture recognition problem in minutes. But it couldn't solve physics.

What Lokkji understood immediately — "น่าจะหยุดก่อน เพราะเหมือนคอมมูบอกว่า Q8 จะมีปัญหาคือช้า" — is exactly right. The machine was telling us something. Good engineers listen to that. We stopped before wasting more time, and now we know: this path requires either a Q4 model, a different (more CPU-friendly) architecture, or a GPU.

The harvest isn't always the crop you planted. Sometimes it's the knowledge of what the soil can and cannot grow.

## Honest Feedback

Three points of friction worth naming:

**1. Architecture opacity upfront.** When we looked at the HuggingFace page, we got parameter counts and quantization options — but not a clear warning that `qwen35moe` with GatedDeltaNet is a fundamentally different inference pattern from standard transformer MoE. If we'd had that context earlier, we might have started with Q4 or a different model family entirely. Model cards could be clearer about CPU inference viability.

**2. Ollama's silent failure mode.** The original error `unable to load model` gave no hint that the problem was an unknown architecture — it just failed. A message like "unknown architecture: qwen35moe, ollama >= 0.21 required" would have saved 15 minutes of diagnosis. This is a pattern: tools that fail silently waste more time than tools that fail loudly with context.

**3. The timeout-as-metric problem.** We ran multiple curl requests with various timeouts trying to measure token speed, but because each one aborted before producing output, we couldn't get actual tok/s numbers. Better approach next time: use `ollama run` interactively in a tmux window and watch the spinner — if no characters appear within 30 seconds, stop and recalibrate. Don't try to measure what isn't moving.

## Lessons Learned

- **Q8_0 on Xeon E5-2690 v4 is not viable** for qwen35moe (GatedDeltaNet hybrid). Memory bandwidth ~100 GB/s / 35 GB model = theoretical 2-3 tok/s, but in practice worse due to SSM sequential computation patterns.
- **Q4_K_M (22 GB) is the next candidate** — half the size means twice the bandwidth headroom, and it fits the SSD easily.
- **Ollama version matters for new architectures** — always check `ollama --version` and compare against model release date.
- **`/mnt/data` (SSD, 1.6 TB free) is the right place for models** — never download large files to root filesystem at 92% capacity.
- **When the machine says stop, stop** — Lokkji's instinct to halt the Q8 experiment early was correct. Don't burn time watching a frozen progress bar.
